# Project Plan

## Overview

This plan breaks the work described in [scope.md](scope.md) into the phases actually followed to build the Texas Tech-themed portfolio website, along with a risk analysis using the TAME framework (Transfer, Accept, Mitigate, Eliminate).

## Work Breakdown Structure (WBS)

| # | Phase | Tasks |
|---|-------|-------|
| 1.0 | Idea & Concept | Settled on a Texas Tech-themed portfolio to reflect pride in being a TTU student and to showcase my work. |
| 2.0 | Scope & Requirements | Reviewed the project scope assignment, the rubric, and the requirements; used these to formulate the plan. |
| 3.0 | Design Direction | Landed on a paper-cartoon, western-inspired look pulling from Texas Tech branding and the EA CFB 27 video game — difficult to adapt into something original rather than a copy. |
| 4.0 | Base Build | Started with barebones HTML structure, then iterated on the CSS design from there. |
| 5.0 | Project Selection & Content | Chose which projects to feature (Fragbase, Green Thumb) and wrote up their descriptions. |
| 6.0 | About Page | Built the About page to showcase the skills gained through schooling, along with education and background. |
| 7.0 | Testing & Polish | Checked the layout across multiple screen sizes, fixed bugs found along the way, refined styling details. |
| 8.0 | Submission | Finalized README.md and documentation, confirmed the repo meets all deliverables, submitted the repository link. |

## Risk Analysis (TAME Framework)

**Transfer** — Hosting the site on GitHub Pages rather than self-hosting. This transfers the risk of uptime, hosting infrastructure, and server maintenance to GitHub instead of it being my responsibility.

**Accept** — Portfolio sites are a somewhat common project idea, and there's a risk this doesn't push the envelope as far as it could have. I'm accepting that risk since the project is far enough along that changing direction now isn't worth it.

**Mitigate** — Tested the design across multiple common screen sizes (mobile, tablet, desktop) to catch layout bugs and keep the design consistent, rather than assuming it would just work everywhere.

**Eliminate** — Kept the project barebones on purpose: no database, no backend, no user accounts. This eliminates the entire risk surface that comes with handling data or authentication, while still meeting what the assignment requires (also would be unessecary).
